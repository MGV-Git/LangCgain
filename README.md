# VPg03 — демо-агент LangChain (погода, контекст пользователя, свои инструменты)

 «по-взрослому» собранный агент на **LangChain** и **LangGraph** с русскоязычным системным промптом, пользовательским контекстом (`user_id`), несколькими инструментами и структурированным ответом. Подходит как шаблон для домашних заданий, прототипов ботов и экспериментов с вызовом внешних API.

---

## Что внутри

| Компонент | Файл | Зачем |
|-----------|------|--------|
| Точка входа, агент, режимы запуска | `main.py` | Сборка модели, инструментов, чекпоинтера, демо и интерактивный чат |
| Схема ответа для модели | `schemas.py` | Pydantic-модель финального ответа; вынесена отдельно осознанно (см. ниже) |
| Зависимости | `requirements.txt` | `langchain`, `langchain-openai`, `langgraph`, `python-dotenv` |

## Особенности и зачем они нужны именно в таких проектах

### 1. Русский язык и консоль Windows

В начале `main.py` для `stdout`/`stderr` включается **UTF-8** (`reconfigure(encoding="utf-8")`). Без этого в типичной консоли Windows (например cp1251) кириллица и смайлики из ответов модели часто превращаются в «кракозябры».

**Где пригодится:** любые ваши скрипты из этой же папки «ОБУЧЕНИЕ», где вы печатаете в консоль результаты LLM, логи на русском или JSON с Unicode.

### 2. Переменные окружения и `.env`

Используется `python-dotenv`: ключ API и опционально базовый URL подгружаются из окружения.

**Где пригодится:** вы не коммитите секреты в репозиторий; один и тот же код можно запускать дома, в университете и на другой машине, подставляя только `.env`. Для курсовых и пет-проектов это стандартная практика.

Нужно задать как минимум:

- `OPENAI_API_KEY` — обязательно  
- `OPENAI_BASE_URL` — по желанию (совместимые прокси и альтернативные эндпоинты)

### 3. `init_chat_model` и модель `openai:gpt-4o-mini`

Модель инициализируется через унифицированный API LangChain с указанием провайдера в строке. Параметры `max_tokens` и `temperature` заданы явно — поведение предсказуемее для демо и отчётов.

**Где пригодится:** когда вы меняете модель в одном месте (для сравнения качества/стоимости в разных вариантах заданий) без переписывания всего кода.

### 4. Инструменты (`@tool`): погода, «геолокация», туристический совет

Реализованы три функции:

- **`get_weather_for_location`** — демо-данные погоды по названию города (без реального API).  
- **`get_user_location`** — по `user_id` из **контекста выполнения** возвращает условный регион (демо: «Флорида» для `user_id == "1"`, иначе «Сан-Франциско»). Показывает паттерн «модель не угадывает профиль пользователя — ей даёт сервер/рантайм».  
- **`get_travel_tip`** — короткие подсказки по региону/городу (русские и латинские ключи в словаре).

**Где пригодится в ваших проектах:**

- Заменить заглушку погоды на **Open-Meteo**, WeatherAPI и т.д. — тот же интерфейс tool.  
- Подставить **реальную БД или CRM**: `get_user_location` превращается в запрос профиля по ID.  
- Любой новый сценарий (расписание, курсы валют, поиск по вашим файлам) — отдельный `@tool`, агент сам решает, когда его вызвать.

### 5. Контекст `Context` и `ToolRuntime`

`Context` — это `dataclass` с полем `user_id`. Инструмент `get_user_location` принимает `ToolRuntime[Context]` и читает `runtime.context.user_id`.

**Где пригодится:** это основа для **мультипользовательских** ботов и сценариев «ответ зависит от того, кто спрашивает»: права доступа, язык, город по умолчанию, подписка — всё можно прокидывать через контекст, не смешивая это с текстом промпта.

### 6. Структурированный ответ: `ToolStrategy(ResponseFormat)`

Финальный ответ приводится к схеме **Pydantic** (`ResponseFormat` с полем `text`). Пользователю в консоль выводится только `sr.text`, без сырого словаря и `repr`.

**Где пригодится:**

- В отчётах и демо — аккуратный, один тип вывода.  
- Дальше можно добавить поля (категория, уверенность, ссылки) без ломания парсинга «голого» текста.  
- Удобно стыковать с **API** (FastAPI возвращает ту же схему JSON).

### 7. Схема в отдельном файле `schemas.py`

В комментарии к коду указано важное: класс ответа лежит не в `__main__`, а в модуле `schemas`, чтобы в **чекпоинте** LangGraph был стабильный путь типа `schemas.ResponseFormat`. Иначе при запуске «как скрипт» и «как модуль» путь к классу различается, что мешает десериализации.

**Где пригодится:** как только вы добавите сохранение состояния диалога, фоновые задачи или перенос чекпоинтов между запусками — такой приём избавляет от тонких ошибок сериализации.

### 8. Чекпоинтер `InMemorySaver` и `JsonPlusSerializer`

Включён in-memory чекпоинтер с явным **allowlist** для типа `ResponseFormat` (объединение с `SAFE_MSGPACK_TYPES`), чтобы не получать предупреждения LangGraph про незарегистрированные типы при десериализации.

**Где пригодится:**

- **Демо и обучение:** диалог с `thread_id` помнится в рамках процесса — видно, как агент опирается на историю (например, ответ «Спасибо!» после вопроса о погоде).  
- Для «боевых» проектов память обычно переносят на **персистентный** saver (БД, Redis); паттерн с `thread_id` и контекстом остаётся тем же.

### 9. Системный промпт на русском

Агент жёстко настроен как русскоязычный синоптик: даже если пользователь пишет по-английски, ответ — по-русски; топонимы — в русской форме («Флорида», не Florida).

**Где пригодится:** единый стиль для курсовых работ, презентаций и проверки преподавателем; шаблон можно скопировать и поменять роль (преподаватель, техподдержка, гид).

## Режимы запуска

После установки зависимостей и настройки `.env`:

```bash
python main.py          # режим demo: два заранее заданных запроса
python main.py demo     # то же
python main.py chat     # интерактивный ввод в консоли
```

В режиме **chat** идентификатор пользователя берётся из переменной **`DEMO_USER_ID`** (если не задана — используется `"1"`). Так можно быстро проверить разное поведение `get_user_location` без правки кода.

## Ограничения (честно для учебного проекта)

- Погода и «геолокация» — **демо-заглушки**, не реальные сервисы.  
- Память диалога живёт **только в RAM** до завершения процесса.  
- Стоимость и лимиты запросов к OpenAI — на вашей стороне.

## Куда развивать дальше в рамках ваших же учебных и пет-проектов

1. Подключить **реальный API погоды** и кэширование.  
2. Хранить пользователей в **SQLite/PostgreSQL** и подставлять профиль в `Context`.  
3. Обернуть агента в **FastAPI** или Telegram-бота — те же tools и схема ответа.  
4. Заменить `InMemorySaver` на персистентный чекпоинтер, если нужны длинные сессии.  
5. Расширить `ResponseFormat` (например, поле `sources` или `language`) для более строгой сдачи заданий с формальным форматом вывода.

Этот репозиторий — компактная база: в нём уже собраны приёмы, которые встречаются в продвинутых туториалах по LangChain/LangGraph, но при этом код остаётся читаемым для пошагового разбора на занятиях.
